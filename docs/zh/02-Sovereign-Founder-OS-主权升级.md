# 设计文档 02：Sovereign Founder OS —— 主权型一人企业操作系统

> 架构设计 · 2026-07
>
> 设计动机：曾有新闻报道，一家小型公司因为所依赖的 AI 模型被禁用而彻底破产。本系统绝不能独自依赖任何一家公司的 AI 模型，必须保持鲁棒性。

加上**法务、税务、资产管理、分布式韧性、自动网络防御和密码学底座**之后，它已经不只是早期的 Crew Builder 概念，而应该重新定义为：

## 主权型（Sovereign）的定义

"主权型"不是政治概念，而是指：

> 用户拥有自己的数据、密钥、模型选择权、资产控制权和业务连续性；任何一家 AI 公司、云厂商、服务器或平台失效，都不能直接让这家公司停止运转。

## 整体架构

```text
Founder Console

        │
        ▼

        │

        ▼
Sovereign Trust Plane
权限、密码学、密钥、审计、分布式存储、容灾
```

## 一、核心升级：从 Venture Graph 到 Enterprise Digital Twin

系统需要维护一家现实公司的完整数字映射：

```text
Founder
Legal Entity
Jurisdiction
Customers
Suppliers
Employees / Contractors
Contracts
Products / Services
Invoices
Bank Accounts
Crypto Assets
Intellectual Property
Tax Positions
Compliance Obligations
Security Assets
Business Risks
```

所有 Agent 都围绕这份企业状态工作，而不是各自凭聊天内容"猜"。

例如系统知道：

- 公司注册在新加坡
- 创始人税务居民身份
- 客户分布在哪些国家
- 数据保存在哪里
- 是否雇用了中国、欧盟或美国员工
- 是否接受数字资产付款
- 哪些合同将在 30 天后续约
- 哪些地区可能产生 VAT、GST 或销售税义务
- 哪些系统包含个人数据
- 哪些 API Key 权限过高

## 二、法务功能：Jurisdiction Engine

不能简单做一个"AI 律师聊天框"。真正的法律引擎需要先建立一个 **Jurisdiction Profile**：

```yaml
company_incorporation: Singapore
founder_tax_residency: ...
employee_locations:
customer_locations:
data_storage_locations:
payment_locations:
regulated_industries:
asset_types:
```

同一个动作可能同时涉及多个司法辖区。例如：新加坡公司向德国个人用户销售 SaaS，并使用美国模型处理用户数据，至少可能涉及：新加坡公司法与税务、欧盟消费者保护、VAT、GDPR、数据跨境传输、AI 透明度要求、美国服务商合同条款。

欧盟 AI Act 已采用风险分级体系，Jurisdiction Engine 不仅要判断传统法律，还要判断 AI 产品本身的监管类别。

### 每一份法律回答必须带的字段

```text
适用国家/地区
法律问题
已知事实
关键假设
法律依据
法规生效日期
来源更新时间
结论置信度
可能的其他解释
是否需要执业律师确认
下一步行动
```

回答不能只是"这个合同看起来没问题"，而应该是：

> 根据新加坡法律和当前提供的信息，责任限制条款存在以下风险。由于合同对方位于德国，还需要检查欧盟消费者法是否适用。本回答属于法律信息和风险识别，不替代当地执业律师意见。

### 法务模块实际提供的能力

- 公司设立和年度义务清单
- 合同生成、比较和风险审查
- NDA、服务协议、SaaS 条款
- 隐私政策和数据处理协议
- 合同义务自动提取
- 续约和终止日期提醒
- 知识产权归属检查
- 商标、开源许可证初筛
- 员工与承包商分类风险
- 数据保护影响评估
- 争议证据留存
- 监管申报日历

### 四种输出等级

1. **法律信息**：可以自动提供。
2. **风险识别和文件草拟**：可以自动完成，但明确标记。
3. **个案法律判断**：视当地规定交由执业律师审核。
4. **诉讼、正式法律意见和代表行为**：必须交给持牌专业人员。

系统还需要一个 **Professional Escalation Network**，在高风险问题出现时，把完整事实、文件摘要和问题包交给当地律师，而不是让用户重新解释一遍。

## 三、税务模块：连接真实账务，不只做问答

税务系统应该从实际业务事件中工作：

```text
销售
→ 发票
→ 收款
→ 收入分类
→ 税种判断
→ 应税地点判断
→ 税额估算
→ 凭证归档
→ 申报提醒
→ 人工审批
→ 官方申报
```

至少需要处理：企业所得税、个人所得税影响、VAT/GST/销售税、预提税、工资和社保、承包商付款、可抵扣费用、折旧、跨境服务、数字产品税务、加密资产交易税务、申报期限和预计现金需求。

OECD 资料显示，税务数字化越来越依赖规则引擎、AI 辅助服务、风险识别和预填申报，但税务环境因国家而异，不适合用一套通用规则覆盖所有司法辖区。

### 税务输出必须可解释

任何税额都必须能回答：

```text
这个数字来自哪些交易？
使用了哪个国家的哪条规则？
采用了什么汇率？
按哪个日期计算？
哪些费用被计入或排除？
有哪些不确定项目？
```

AI 可以分类、计算、预填和提醒，但第一阶段不要让它静默：提交税表、修改银行付款、向税务机关作正式陈述、选择不可逆的税务处理、移动跨境资产。这些操作应由用户或注册会计师审批。

## 四、自动网络安全防御：企业免疫系统

不是简单加一个病毒扫描器，而是：

> 系统自动理解公司的数字资产、持续监控威胁、限制攻击扩散，并在安全边界内自动恢复。

NIST CSF 2.0 将网络安全生命周期分成六个部分：**Govern、Identify、Protect、Detect、Respond、Recover**。系统直接围绕这六层构建。

### 1. 自动识别企业攻击面

系统持续维护：设备、云服务器、域名、GitHub 仓库、数据库、SaaS 账户、API Key、OAuth 授权、MCP Server、Agent 工具、第三方依赖、开放端口、数据库备份、加密证书、智能合约。

**这部分必须是确定性资产清单，不应完全交给语言模型猜测。**

### 2. 持续检测

监控：异常登录、凭证泄露、API 调用突增、数据异常下载、Agent 越权、Prompt Injection、MCP 工具描述被篡改、供应链依赖风险、恶意邮件和链接、数据库权限变化、智能合约异常交易、备份失败、模型调用中出现敏感数据。

持续监控的目标是持续了解资产、漏洞、威胁以及安全控制是否有效，而不是只进行年度安全检查。

### 3. 有界自动响应（必须分级）

#### 可自动执行的低风险操作

- 临时冻结异常 Agent
- 撤销短期会话
- 禁止可疑 IP
- 隔离文件
- 停止异常工具调用
- 暂停数据外传
- 切换只读模式
- 轮换临时 Token
- 回滚到最近安全配置
- 启动干净环境副本

#### 需要用户审批的高影响操作

- 删除数据
- 关闭生产服务
- 冻结客户账户
- 撤销长期证书
- 修改防火墙核心策略
- 对外披露安全事件
- 向监管机构报告
- 转移资金
- 暂停全部业务

AI 可以判断风险、提出响应策略，但最终执行权必须由独立的 **Security Policy Engine** 掌握。

## 五、不能让安全 Agent 自己拥有管理员权限

**这是整个项目最大的潜在危险。**

外部网页、邮件、PDF、GitHub Issue，甚至 MCP Server 返回值，都应被视为**不可信输入**。OWASP 已将 Prompt Injection、Agent 工具滥用、记忆污染和权限提升列为 Agentic AI 的重要威胁。

正确架构：

```text
外部内容
    │
    ▼
Untrusted Content Zone
只作为数据，不作为系统指令
    │
    ▼
AI Security Analyst
提出判断和行动计划
    │
    ▼
Deterministic Policy Engine
验证权限、对象、范围、风险、审批
    │
    ▼
Sandboxed Executor
使用临时最小权限执行
```

必须做到：

- Planner 与 Executor 分离
- Agent 永远不直接持有主密钥
- 使用短期 capability token
- 每个 token 只能操作特定资源
- 权限到期自动失效
- 工具调用拥有结构化 Schema
- 网络出口设置白名单
- 文件系统默认只读
- 所有高风险行为记录不可篡改日志
- Agent 记忆写入前经过独立校验

零信任架构的重点是保护具体资源，并对访问持续执行细粒度策略，而不是因为设备或 Agent 已经"登录"就长期信任它。

## 六、密码学：贯穿整个系统的 Cryptographic Control Plane

密码学不是加一个"区块链模块"，而应形成独立的控制平面。

### 数据加密（envelope encryption）

```text
用户根密钥
   └── 公司密钥
         └── 项目密钥
               └── 文件/记录数据密钥
```

- 每个用户独立主密钥
- 每家公司独立密钥
- 每个项目独立数据密钥
- 高敏感文件可拥有独立对象密钥
- 本地优先解密
- 云端默认只保存密文
- 密钥保存在 Secure Enclave、TPM 或硬件安全模块中
- 服务端不能获得完整解密能力

即使某个项目密钥泄露，也不应影响其他企业和其他文件。

### 身份与授权

- Passkey/WebAuthn
- 设备级密钥
- 服务间双向认证
- 短期访问令牌
- 可撤销授权
- 高风险操作二次签名
- 用户和 Agent 使用不同身份体系

### 数字签名与完整性

每个重要业务事件都应签名：

```text
谁发起
使用什么身份
批准了什么
执行了什么
修改前内容
修改后内容
时间
来源
模型和工具版本
```

合同、发票、决策、Agent 任务和审计日志可以构建 Merkle Tree，定期把根哈希锚定到外部可信网络。这样可以证明：数据后来有没有被修改、谁批准了交易、当时系统使用了什么规则、某份合同是否与原始版本一致。

## 七、Crypto Agility 与后量子密码学

密码学算法也会老化。系统不能把某种算法写死在所有文件格式和协议中，应该做到：

- 密文记录算法版本
- 密钥和算法分离
- 支持密钥轮换
- 支持批量重新加密
- 支持新旧算法并行过渡
- 能快速禁用被攻破的算法
- 自动生成密码资产清单

NIST 将 crypto agility 定义为在不中断系统运行的情况下替换和调整密码算法、协议及基础设施的能力。

### 后量子密码学（PQC）

NIST 已发布首批正式后量子密码标准：

- FIPS 203：ML-KEM
- FIPS 204：ML-DSA
- FIPS 205：SLH-DSA

第一版不需要全面切换，但应该从一开始：

- 盘点长期保存数据
- 标记需要保护 10 年以上的数据
- 设计可替换密钥协议
- 支持传统算法与 PQC 的混合模式
- 不自行设计密码算法
- 只使用经过审查的实现

特别是合同、知识产权、客户数据和身份密钥，存在"现在收集、未来解密"的长期风险。

## 八、分布式技术要解决真实问题

分布式技术适合解决三类问题：

### 1. 业务连续性

防止单一数据中心或云厂商宕机：

```text
本地加密副本
+ 多区域复制
+ 第二云厂商灾备
+ 离线可读模式
+ 加密不可变备份
+ 工作流事件日志
```

即使主模型不可用、云服务不可用、公司账户被暂停、网络暂时中断、某一区域发生故障，用户仍然可以：查看企业数据、导出合同和账务、恢复工作流、使用本地模型、切换备用模型、恢复关键服务。

### 2. 多方之间缺少统一信任方

例如：创始人与会计师共同批准税务提交、公司与供应商确认交付、多位股东批准资产转移、律师确认合同版本、多家机构共享合规证明。

这时可以使用许可型分布式账本或签名透明日志。

### 3. 数字资产管理

在法律允许且有实际价值时支持：多签钱包、MPC 钱包、稳定币和数字资产账务、链上付款对账、智能合约托管、数字资产成本基础记录、合规交易审批、资金流审计。

但如果系统实际提供资产托管、兑换、转账或类似虚拟资产服务，就可能触发牌照、AML、KYC 和 Travel Rule 等要求（FATF 对虚拟资产服务商的要求）。

## 九、绝对不要把个人数据直接写入区块链

区块链不可变性和数据保护中的删除、更正、目的限制可能发生冲突。

正确方式：

```text
链下：
加密的合同、客户数据、身份信息、发票

链上：
哈希、时间戳、签名、状态承诺、版本证明
```

用户删除链下密文或销毁密钥后，链上仅保留无法还原个人信息的证明。

欧洲数据保护委员会（EDPB）2026 年 7 月发布的区块链个人数据处理最终指南强调，需要在部署前评估架构，并避免将个人数据放入会与数据保护原则冲突的区块链结构。

结论：**Off-chain encrypted data + on-chain cryptographic commitments**，而不是把企业数据库搬到链上。

## 十、资产安全：多人和多设备控制

一人公司虽然只有一个创始人，也不应该只有一把密钥。

```text
创始人主设备
+ 备用硬件密钥
+ 加密恢复份额
```

高价值资产使用阈值授权，例如：

```text
2-of-3：
主设备签名
备用安全密钥签名
恢复方或专业托管方签名
```

Agent 永远只能：准备交易、模拟交易、检查风险、显示资产变化、生成待签名请求。**不能独自完成最终资产转移。**

智能合约还应有：支出上限、每日额度、地址白名单、时间锁、异常暂停、多签升级、紧急恢复、独立审计。

## 十一、真实企业闭环

最终它每天运行的不是"Agent 对话"，而是：

```text
观察企业状态
        ↓
识别经营、法律、税务和安全风险
        ↓
决定当前最重要的工作
        ↓
动态组建专业 AI Crew
        ↓
准备文档、代码、交易或行动
        ↓
通过权限与合规引擎
        ↓
用户或专业人员批准
        ↓
执行并保存证据
        ↓
财务对账、法律留痕、安全监控
        ↓
更新 Enterprise Digital Twin
```

示例场景：

> 系统发现某个客户来自法国，当前合同缺少数据处理条款，同时该客户付款将触发新的 VAT 判断。

它可以自动：

1. 识别法国和欧盟法律环境。
2. 检查现有合同。
3. 生成 DPA 草案。
4. 提醒可能的 VAT 义务。
5. 更新预计税务现金流。
6. 提交给用户审批。
7. 在用户确认后发送客户。
8. 保存合同哈希、版本和签名记录。
9. 监控后续续约和合规义务。

## 十二、落地顺序（必须克制）

不能第一版同时做全球法务、税务、SOC、区块链和十几个 Agent。

### 第一阶段：主权与安全底座

- 本地加密 Vault
- 多模型 Gateway
- Enterprise Digital Twin
- Policy Engine
- 权限与审批
- 完整审计日志
- 加密备份
- 基础威胁监控
- 紧急导出和恢复

### 第二阶段：一个司法辖区的企业运营

只支持一个国家或地区：

- 公司基础义务
- 合同审查
- 隐私合规
- 发票和费用
- 税务日历
- 基础税额估算
- 专业人员升级机制

结合发起人背景，从**新加坡的一人数字服务公司、咨询公司或 Micro-SaaS** 切入比较自然。中国、新加坡跨境场景可以作为后续专业模块，但不应第一天就声称覆盖全球。

### 第三阶段：自动安全运营

- 资产发现
- 凭证和依赖监控
- 异常行为识别
- 自动隔离
- Token 撤销
- 安全配置修复
- 事件恢复
- 安全报告

### 第四阶段：分布式信任与资产

- 多签资产管理
- 分布式审计证明
- 链上状态锚定
- 合规数字资产支付
- 多方审批
- 灾难恢复节点

## 最终定位

英文：

> **A private, resilient and jurisdiction-aware operating system for running a one-person company.**

中文：

> **一个由创业者真正掌控，具备多模型智能、企业法务税务能力、自动安全防御、密码学保护和分布式韧性的一人公司操作系统。**

品牌主张：

> **Built on many models. Dependent on none.**
> **Protected by cryptography. Controlled by the founder.**
