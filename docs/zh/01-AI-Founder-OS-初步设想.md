# 设计文档 01：AI Founder OS —— 面向小白的一人创业操作系统

> 产品设想 · 2026-07
> 早期概念：Crew Builder（现统一命名为内部组件 **Crew Orchestrator**）

## 产品定位

不是让用户自己创建"市场 Agent、程序员 Agent、销售 Agent"，而是用户只需要说：

> "我想利用自己的区块链和后端经验，做一个面向中小企业的 AI 产品，但不知道从哪里开始。"

系统便自动完成：

**需求澄清 → 商业机会分析 → 创业路线生成 → 动态组建 AI 团队 → 执行任务 → 产出交付物 → 收集市场反馈 → 调整战略**

用户面对的是"下一步该做什么"，而不是复杂的 Agent 参数。

---

## 一、核心抽象：Venture Graph（创业状态图谱）

这个系统最重要的东西不是 Agent，而是一个长期存在的 **Venture Graph**。

它记录的不是聊天历史，而是一家创业项目当前的真实状态：

| 对象 | 内容 |
| --- | --- |
| Founder Profile | 技能、资金、时间、风险偏好、资源 |
| Customer | 目标客户、用户画像、痛点 |
| Problem | 要解决的问题及证据 |
| Offer | 产品、服务、价格和价值主张 |
| Assumption | 尚未验证的商业假设 |
| Experiment | 访谈、广告、落地页、预售测试 |
| Artifact | 商业计划、页面、邮件、代码、合同 |
| Metric | 转化率、获客成本、收入、留存 |
| Commitment | 已承诺事项、截止日期、客户交付 |
| Policy | 用户允许 AI 做什么、不允许做什么 |

**每次 AI 行动都必须改变这个状态图谱**，而不是仅仅生成一段看起来很聪明的回答。

这会让它从"创业聊天机器人"变成真正的创业操作系统。

---

## 二、小白真正看到的产品形态：Founder Cockpit

首页不应该是 Agent 聊天室，而应该是一个非常简单的 Founder Cockpit。

用户看到四个东西：

1. **你现在在哪里**
2. **目前最大的未知是什么**
3. **下一步最值得做什么**
4. **AI 已经为你准备好了什么**

例如：

> 当前阶段：问题验证
> 最大风险：尚未证明客户愿意付费
> 推荐行动：联系 10 位潜在客户
> AI 已准备：访谈问题、目标名单、邀请邮件、记录模板

用户点击"开始"，系统才在内部组建临时 Crew。

### Agent 不应该长期固定

不要永久创建一个"市场总监 Agent"。更合理的是根据任务临时生成：

- Researcher：寻找证据
- Strategist：形成假设
- Builder：制作交付物
- Critic：寻找漏洞
- Operator：执行获批操作
- Evaluator：判断结果是否有效

任务结束后，这支 Crew 就解散，只保留结果、证据和决策记录。

这比让多个 Agent 永久互相聊天更便宜、更稳定，也更容易控制。

## 三、整体系统架构

```text
用户创业目标
      │
      ▼
Mission Compiler
将自然语言转成结构化创业任务
      │
      ▼
Venture Graph
客户、问题、产品、证据、实验、指标
      │
      ▼
Crew Orchestrator
根据当前瓶颈动态组队
      │
      ▼
Policy & Privacy Gateway
判断数据级别、权限、成本和审批要求
      │
      ▼
┌──────────────┬──────────────┬──────────────┐
│ 本地模型      │ 云端模型      │ 工具与外部 Agent │
│ 私密处理      │ 通用推理      │ MCP / A2A       │
└──────────────┴──────────────┴──────────────┘
      │
      ▼
Sandboxed Execution
浏览器、代码、邮件、文档、数据分析
      │
      ▼
Critic + Evaluator
事实核查、质量评分、安全检查
      │
      ▼
Artifact Ledger
保存交付物、来源、版本、决策和审批记录
```

## 四、应该吸收的前沿思想

### 1. Dynamic Agent Composition

Crew 不是用户手动配置，而是由任务编译器根据以下因素动态生成：

- 当前创业阶段
- 任务类型
- 所需工具
- 数据敏感度
- 成本预算
- 错误风险
- 是否需要人类审批

这才是 Crew Orchestrator 真正有价值的部分。

### 2. MCP 作为工具接口

邮件、日历、GitHub、CRM、文件、浏览器和数据库，可以逐步通过 MCP 接入。MCP 的定位是让模型应用以统一方式调用外部工具和数据源。

但每个 MCP 工具必须有独立权限，不能因为用户授权了 Gmail，就允许所有 Agent 随意读取或发送邮件。

### 3. A2A 作为外部 Agent 协议

A2A 更适合未来让 Founder OS 与外部专业 Agent 协作，例如财务 Agent、设计 Agent、法律服务 Agent 或其他公司的 Agent。它强调 Agent 发现、能力描述、任务委派和跨系统协作。

**第一版不需要实现 A2A，预留接口即可。**

### 4. Durable Agent Workflow

创业任务可能持续数天甚至数月，不能靠一次聊天调用维持状态。

每项工作都应该是可暂停、恢复、重试和回滚的持久化工作流。Temporal 一类系统通过事件历史记录工作流进度，并在故障后恢复执行，这种思想非常适合长期 Agent 任务。

### 5. Model Routing

不要绑定一个模型。建立统一 Model Gateway，根据任务选择：

- 本地小模型：隐私分类、信息抽取、敏感内容总结
- 便宜云模型：格式化、普通文案
- 强推理模型：战略分析、复杂研究
- 多模态模型：网页、图片、文档和视频
- 编码模型：网站和原型开发

每次调用都记录成本、延迟、质量和发送了哪些数据。

### 6. Agent Improvement Loop

每次任务都保存：

- 输入状态
- 计划
- 模型调用
- 工具调用
- 交付结果
- 用户是否接受
- 最终业务指标

之后用 traces、evals 和真实反馈评估哪些流程有效，而不是靠主观感觉修改 Prompt。

## 五、个人数据安全设计

首先必须承认：

> **不存在"绝对不会泄露"的联网系统。**

产品可以追求的是：

> 默认本地、服务器看不到明文、最小化数据暴露、权限极小、所有危险行动可审计和可撤销。

不要宣传"绝对安全"，而可以宣传：**Local-first、end-to-end encrypted、server-blind by default。**

Local-first 的核心是让数据首先存在于用户设备，用户拥有实际控制权，而不是只能通过厂商服务器访问自己的数据。

### 三层隐私区

#### 红区：禁止离开设备

- 身份证件
- 客户完整名单
- 银行和财务数据
- API Key
- 私人邮件全文
- 合同原件
- 未发布商业秘密

红区任务只允许本地模型和本地工具处理。

#### 黄区：处理后才可发送

- 脱敏客户需求
- 匿名化业务数据
- 删除姓名后的邮件内容
- 只保留必要字段的文档片段

本地 Privacy Gateway 先完成脱敏，再让用户看到"即将发送给模型的内容"。

#### 绿区：可以使用云端模型

- 公开市场资料
- 通用文案要求
- 已发布产品信息
- 无个人身份信息的研究问题

### 安全底座的非谈判项

#### 本地加密保险库

所有创业数据默认存储在本地加密 Vault 中：

- 每个项目独立数据密钥
- 密钥存放在系统安全区域
- 云端只保存密文
- 服务端永远不持有用户主解密密钥
- 加密备份由用户恢复密钥控制

多设备同步应该是端到端加密同步，而不是服务器解密后再同步。

#### Passkey 登录

账户认证使用 WebAuthn/passkey，而不是只依赖密码。但登录密钥与数据加密密钥应分开设计，避免身份系统被攻破后直接导致数据解密。

#### Zero Trust 权限

任何 Agent、插件和工具都默认不可信。每次授权应包含：

```text
谁可以调用
可以调用什么
可以读取哪些字段
可以操作哪个项目
有效多长时间
最多调用几次
是否需要人工批准
```

#### 高风险动作必须审批

以下操作永远不能静默执行：

- 发邮件或私信
- 发布网页和社交媒体内容
- 删除文件
- 部署代码
- 修改生产数据
- 签署合同
- 下单和付款
- 创建广告支出
- 向客户作出承诺

系统可以生成预览和 diff，但最终必须由用户签名确认。

四级自动化：

| 等级 | 能力 |
| --- | --- |
| L0 建议 | 只提供建议 |
| L1 草稿 | 生成内容但不执行 |
| L2 审批执行 | 用户确认后执行 |
| L3 有界自动化 | 在预算和权限范围内自动运行 |

财务、法律和不可逆操作最多停留在 L2。

#### 工具沙箱

代码执行、网页访问和第三方插件应放在隔离环境中：

- 默认无网络
- 网络域名白名单
- 只读文件挂载
- 临时凭证
- 限制 CPU、内存和时间
- 任务完成后销毁环境

Prompt Injection 不能只靠系统提示词解决。外部网页、邮件和文档可能包含恶意指令，并诱导 Agent 滥用工具或泄露数据。OWASP 将间接 Prompt Injection、工具滥用、权限提升和数据外泄列为 Agent 系统的重要风险。

因此必须把"模型建议做什么"和"系统是否允许做"彻底分开。

#### Confidential Computing 作为高级云能力

当本地设备算力不足时，可以提供可选的机密云推理：

- 在可信执行环境内处理
- 远程证明运行的是指定代码
- 数据只在受保护环境中解密
- 运行后不持久化明文

Apple Private Cloud Compute 的设计提供了值得参考的方向：设备端作为隐私策略根，在需要更强算力时才把请求交给受验证的云计算环境。

**这适合后期做，不适合第一版自行实现。**

## 六、第一版应该做什么

不要一开始做"支持所有行业的一人公司"。首先只支持：

> **数字服务、咨询业务和 Micro-SaaS 创业者。**

这三类业务更容易形成完整闭环，也更适合自动化。

### 第一版只做五条工作流

#### 1. Idea to Opportunity

用户描述技能和想法，系统输出：目标客户、问题假设、替代方案、差异化、主要风险、下一步验证计划。

#### 2. Customer Validation

系统帮助用户：制定访谈问题、整理潜在客户名单、生成邀请邮件、总结访谈、提取共同痛点、更新商业假设。

#### 3. Offer Builder

生成：产品定位、服务范围、定价结构、价值主张、FAQ、销售页面文案。

#### 4. Launch Kit

生成并管理：落地页、邮件序列、社交内容、演示脚本、产品介绍、发布清单。

#### 5. Founder Daily Board

每天只告诉用户：

- 今天最重要的三件事
- AI 可以代做哪些
- 哪些需要用户决策
- 当前现金、客户和实验状态
- 哪个假设最危险

### 第一版明确不做

自动付款、自动签约、自动投放广告、自由浏览用户全部文件、未审核的 Agent 市场。

## 七、技术路线

（背景：发起人 Java 后端基础强，需要进入 AI 工程生态）

### 本地 AI Runtime

- Python
- FastAPI
- 类型化状态模型
- Agent workflow
- 本地模型调用（可通过 llama.cpp 一类运行时接入）
- 文档处理
- MCP client
- 隐私分类和脱敏
- 沙箱任务执行

### 用户界面

- React / TypeScript
- 最初可以做 Web UI
- 需要真正 local-first 时封装成 Tauri 桌面应用
- 所有敏感数据访问通过本地 runtime

### 云端 Control Plane

第一版甚至可以没有。后期再使用 Spring Boot 实现：

- 账户与订阅
- 加密同步协调
- 设备管理
- 模型额度
- 插件签名与分发
- 匿名质量指标
- 团队和企业权限

**云端 Control Plane 不碰用户明文业务数据。**

### 工作流引擎

MVP 先实现明确的 typed state machine，不要一开始上十几个 Agent 框架。

规模扩大后，再引入 Temporal 负责：长期任务、重试、暂停和恢复、定时任务、人工审批、失败补偿。可以继续使用 Java SDK，把可靠工作流部分放在最擅长的技术栈中。

## 八、商业模式

商业模式应与隐私定位一致：

- **免费或开源的本地核心**
- 付费加密多设备同步
- 付费高级模型路由
- 付费机密云推理
- 付费专业创业模板
- 经过审核的工具和技能市场
- 企业私有部署

不要靠分析用户创业数据卖广告，也不要让商业模式依赖收集用户信息。

差异化主张：

> 其他 AI 创业工具是在云端替你做事；
> 你的系统是在用户控制的数据边界内，帮助他经营一家真正属于自己的公司。

## 九、最合适的产品路线

**Crew Orchestrator 是内部技术组件，不是产品名称或第二主品牌。**

最终产品应当表现为：

> **一个懂用户、拥有长期创业记忆、能够动态组建 AI 专家团队，并在严格权限边界内帮助普通人完成创业全过程的私人 Founder OS。**

第一步不是开发十个 Agent，而是先完成这条最小闭环：

```text
用户输入创业想法
→ 形成 Venture Graph
→ 找出最大未知
→ 生成验证实验
→ 产出访谈与落地页材料
→ 收集反馈
→ 更新判断
→ 给出下一步行动
```

只要这条闭环真正可靠，就已经比绝大多数"多 Agent 聊天室"更接近一个实际可用的创业系统。
