# 设计文档 03：Sovereign Founder OS 开源项目企划书 v0.1

> **历史资料：** 本文保留项目早期思路，不是当前规范。若有冲突，以根目录核心文档、RFC 和当前实现为准。

> 项目企划 · 2026-07
>
> 核心结论：**Sovereign Founder OS 是帮助普通人创建和经营一人公司的完整产品。工程上先完成 Sovereign Trust Layer 的最小可信基础，再通过真实创业工作流逐步扩展；开发顺序不改变完整产品定位。**

## 一、项目摘要

### 1.1 项目暂定定位

**Sovereign Founder OS** 是一套帮助任何人创建并经营一人公司的 AI 操作系统，同时让数据、决策权和企业持续运营能力始终掌握在创业者自己手中。

它面向普通人、一人创业者、自由职业者、独立开发者、小型咨询公司、小型企业和 Micro-SaaS 创业者。用户不需要先学会创业方法、软件开发或 Agent 配置，系统应当把模糊起点转化为可执行、可验证、可持续改进的经营路径。

完整产品愿景将帮助普通用户完成：商业机会分析、产品与服务设计、客户开发、合同及企业法务辅助、税务和财务管理、营销和销售执行、软件及网站建设、网络安全监控、数据与数字资产管理、企业连续性与灾难恢复。

安全、隐私、多模型和分布式韧性是底层设计宗旨与差异化护城河，而不是用户每天打开产品的主要目的。它们的设计目标是：

> 即使某一家模型、云厂商、设备、数据库、插件、服务器甚至项目官方服务失效，用户依然拥有自己的数据、密钥、业务状态和继续经营公司的能力。

### 1.2 核心品牌主张

> **Build and run your one-person company with AI—without giving up control of your data, decisions or business.**

中文：

> **用 AI 建立和经营属于你自己的一人公司，同时不放弃对数据、决策和企业的控制权。**

底层承诺仍然是：**Your company. Your data. Your keys. Many models. No master. No single point of failure.**

### 1.3 完整产品结构

#### 产品表层：Sovereign Founder OS

用户看到并使用七个主要模块：

- Venture Studio
- AI Crew
- Product & Delivery
- Customers & Growth
- Finance / Legal / Tax
- Founder Command Center
- Sovereign Trust Layer

#### 中间层：企业运营与 AI 编排

Sovereign Enterprise Graph 保存产品、客户、任务、合同、发票、财务、合规、品牌、知识、指标、风险和假设等权威经营状态。**Crew Orchestrator** 根据业务目标动态组建和约束 AI Crew，用户不需要手动配置 Agent。

#### 底层：Sovereign Trust Layer 与 Sovereign Runtime

Sovereign Trust Layer 负责隐私、安全、多模型、权限、审计、加密、故障恢复、数据主权和业务连续性，由可复用的 **Sovereign Runtime** 承载。其主要组件包括 Model Mesh、Policy Engine、Secure Vault、Audit Ledger、Tool Sandbox 和 Recovery Mesh。

完整产品、仓库和社区的唯一主品牌是 **Sovereign Founder OS**。AI Crew 是用户功能，Crew Orchestrator 是内部编排组件，Sovereign Runtime 是底层技术名称，文档名称统一为 **Sovereign Founder OS Manifesto**。

### 最客观的结论

**值得做，但不能按完整愿景同时开工。**

正确路径（工程顺序，不是产品定位）：

> Sovereign Runtime 安全内核 → 可攻击、可验证的证明 + 一个真实 Founder OS 工作流 → 分布式恢复 → 司法辖区能力 → 更完整的企业运营模块与插件生态

错误路径：

> 同时开发全球法务、全球税务、区块链钱包、自动营销、多 Agent、桌面端、移动端和企业云平台。

## 三、与 OpenClaw 的本质区别

截至 2026 年 7 月 13 日，OpenClaw GitHub 仓库约有 38.3 万 Star 和 8.03 万 Fork。它已经是极其成功的类别定义型项目，因此"超过 OpenClaw"不能被当成普通增长目标，而需要创造一个新的类别。

### 3.1 OpenClaw 已经做得好的地方

OpenClaw 并不是一个完全不重视隐私和安全的项目。它已经具备：

- 自托管和本地状态
- 多聊天渠道
- 持久化记忆
- 多模型支持和模型 fallback
- 工具权限配置
- Docker 工具沙箱
- 插件与 Skill 安全扫描
- 安全审计命令

因此不能把差异简单宣传成 "OpenClaw 在云端，我们在本地" —— 这并不准确。

### 3.2 真正的差异

OpenClaw 的核心产品是：一个可以从各种聊天渠道访问、能够使用工具的个人 AI 助手。其 Gateway 是会话、路由和渠道连接的单一事实来源，插件可以在 Gateway 进程内运行并被要求视为可信代码。基本架构以"强能力助手"为中心。

Sovereign Founder OS 的核心应该是：

> 一个帮助普通人创建和经营一人公司，并且在 Agent、模型、插件或部分基础设施不可信或失效时，仍然保护企业数据、限制行为和恢复业务的操作系统。

| OpenClaw | Sovereign Founder OS |
| --- | --- |
| 个人 AI 助手 | 企业经营与安全操作系统 |
| Gateway 是中心 | 多节点状态与恢复体系 |
| Agent 能力优先 | 企业经营结果优先，由可信边界保障 |
| 插件可作为进程内可信代码 | 插件默认不可信，隔离执行 |
| 聊天和渠道驱动 | 企业状态和工作流驱动 |
| 模型 fallback | 模型、节点、密钥、数据和策略多层 fallback |
| 本地状态 | 加密、分片、复制、可验证的主权数据 |
| 通用任务 | 企业、法务、税务和安全领域任务 |
| 操作者配置权限 | 能力令牌、策略证明和多方制约 |
| 单个助手主动行动 | Planner、Policy、Executor、Auditor 互相分离 |

### 3.3 一句话区别

> OpenClaw 的目标是让 AI 获得一双能做事的手。
> Sovereign Founder OS 的目标是确保这双手永远无法绕过权限、偷走数据、篡改证据或绑架整家公司。

项目不应攻击或贬低 OpenClaw。更好的社区定位是：

> OpenClaw 证明了高自主个人 Agent 的需求；Sovereign Runtime 探索这些 Agent 在真实企业和高敏感环境中如何安全运行。

未来甚至可以提供 OpenClaw Skill 兼容层，将已有生态放入更严格的隔离和权限体系中。

## 四、产品核心概念

### 4.1 Sovereign Enterprise Graph

系统维护的不是一堆聊天记录，而是一家公司的结构化数字孪生：

Founder / Legal Entity / Jurisdiction / Customer / Supplier / Product / Service / Contract / Invoice / Payment / Asset / Intellectual Property / Tax Obligation / Compliance Obligation / Security Asset / Credential / Incident / Business Assumption / Experiment / Metric / Decision / Approval / Artifact

每个 Agent 的工作必须：

1. 读取经过授权的企业状态。
2. 产生结构化计划。
3. 申请执行权限。
4. 产生可验证交付物。
5. 更新企业状态。
6. 留下不可抵赖的操作记录。

### 4.2 Mutually Constrained Autonomy（相互制约的自主性）

项目可以提出并正式定义这个核心思想：

任何单一 Agent 或组件，都不能同时拥有以下全部能力：决定目标、制定计划、授予权限、获取完整数据、执行行动、删除记录、批准结果。

系统拆分为：

| 角色 | 能力 | 明确禁止 |
| --- | --- | --- |
| Planner | 制定计划 | 不持有真实工具凭证 |
| Policy Guard | 判断行动是否允许 | 不生成经营目标 |
| Executor | 执行具体动作 | 不扩大自身权限 |
| Auditor | 校验和记录 | 不执行外部动作 |
| Recovery Controller | 恢复系统 | 不修改正常经营记录 |
| Human Owner | 最终批准 | 高风险操作不可被绕过 |

核心规则：

> 没有任何一个节点既能发起敏感行动，又能批准行动，还能销毁行动证据。

### 4.3 Resilient Trust Mesh

**Resilient Trust Mesh** 是跨节点的总体信任架构；其中 **Recovery Mesh** 专指加密复制、检查点、故障转移和恢复子系统。

整个系统不是依靠一个"万能服务器"，而是形成多个可以互相校验和接替的信任节点：数据节点、密钥节点、执行节点、模型节点、策略节点、审计节点、恢复节点。

"分布式"在这里不等于把所有数据放上区块链，而是：

- 状态存在多个加密副本
- 权限不存在单一控制者
- 核心服务具备替代实现
- 故障可以自动检测
- 任务可以从检查点恢复
- 节点输出可由其他节点验证
- 单个节点损坏不会扩大为全局损坏

```text
              │
              ▼
       Workflow Planner
              │
       只产生计划，不执行
              ▼
┌──────────────────────────────────┐
│       Policy & Trust Plane       │
│ 数据分类 / 权限 / 风险 / 审批 / 法域 │
└──────────────────────────────────┘
              │
              ▼
     Capability Token Issuer
生成短期、限范围、可撤销的执行令牌
              │
              ▼
┌─────────────┬─────────────┬─────────────┐
│ Model Mesh  │ Tool Sandbox│ Domain Packs│
│ 多供应商模型 │ 隔离工具执行  │ 法务税务业务  │
└─────────────┴─────────────┴─────────────┘
              │
              ▼
       Verification Layer
规则校验 / 第二模型复核 / 事实来源 / Schema
              │
              ▼
      Signed Event Ledger
              │
              ▼
 Encrypted Replication & Recovery Mesh
```

系统分为六个平面：

1. **Intelligence Plane**：模型、Agent、规划和推理。
2. **Policy Plane**：确定性权限、风险和审批。
3. **Execution Plane**：工具、浏览器、文件和代码执行。
4. **Data Plane**：加密企业状态和本地存储。
5. **Trust Plane**：身份、密钥、签名、审计和软件来源。
6. **Recovery Plane**：复制、检查点、故障接替和灾难恢复。

## 六、分布式与故障接替设计

### 6.1 单点故障清单（Single Point of Failure Registry）

项目需要维护一份公开的 SPOF Registry：

| 单点风险 | 对策 |
| --- | --- |
| 单一 AI 模型 | 多供应商路由和本地模型 |
| 单一模型账号 | 独立账号、独立供应商、本地降级 |
| 单一设备 | 加密副本和恢复设备 |
| 单一云厂商 | 本地节点、第二存储目标、可导出备份 |
| 单一数据库 | 事件日志、快照、可验证副本 |
| 单一密钥 | 多设备恢复和密钥份额 |
| 单一插件市场 | 可镜像、可离线安装、签名验证 |
| 单一官方服务器 | 核心功能完全自托管 |
| 单一维护者 | 多维护者审批和发布签名 |
| 单一法律数据源 | 多来源、版本和人工复核 |
| 单一安全 Agent | 规则、检测器和人工共同决策 |

### 6.2 事件溯源

权威业务状态通过签名事件构建：

```text
event_id
venture_id
actor_id
action
resource
capability_id
timestamp
payload_hash
previous_event_hash
device_signature
policy_decision_hash
```

每次变更追加事件，不直接悄悄覆盖旧状态。系统定期生成快照，但任何快照都必须能够追溯到签名事件链。

优势：节点可以重新构建状态、篡改可被检测、Agent 不能删除历史痕迹、故障后可以从检查点恢复、可以证明某项决定当时依据了什么信息。

### 6.3 数据复制等级

- **Level 0：单设备模式** —— 只用于开发，不应作为生产推荐。
- **Level 1：本地加密备份** —— 主设备加一个用户控制的加密备份位置。
- **Level 2：多设备复制** —— 桌面端、家庭服务器或 NAS、恢复设备保存加密副本。
- **Level 3：盲云节点** —— 可选云节点只持有密文，不持有解密主密钥。
- **Level 4：高保障模式** —— 高价值操作要求多个信任节点共同批准。

### 6.4 普通状态和权威状态分开

#### 可最终一致的数据

草稿、笔记、非正式研究、营销创意、待办事项。使用本地优先同步和冲突合并。

#### 必须严格控制的数据

已签合同、发票、税务提交、权限变更、资金转移、密钥轮换、安全事件、对外承诺。

这些操作必须使用：明确的写入权、Leader lease、Fencing token、幂等键、版本检查、必要时的多节点批准。

**不能让两个离线节点同时签发互相冲突的资金或权限操作。**

### 6.5 自动接替流程

```text
主节点心跳丢失
      ↓
确认 lease 已到期
      ↓
其他节点验证最近签名事件
      ↓
选择拥有最新有效状态的候选节点
      ↓
签发新的 fencing token
      ↓
旧节点的写入权限失效
      ↓
恢复未完成工作流
```

不能简单地"谁先上线谁就是主节点"，否则可能出现 split-brain。

### 6.6 高风险操作的法定人数

可配置的批准策略：

```text
一般邮件：用户一次批准
生产部署：用户批准 + Policy Guard
资金转移：用户硬件密钥 + 第二恢复设备
核心密钥轮换：2-of-3 恢复份额
高风险合同：用户 + 法务专业人员
```

一人公司并不意味着所有权限必须集中在一台设备和一个密钥中。

## 七、隐私与密码学设计

### 7.1 不能宣传"绝对安全"

公开宣传应使用：Privacy by design、Local-first、Zero-knowledge sync where applicable、Server-blind、Secure by default、Verifiable security、No single point of failure。

不应宣传：永不泄露、绝对安全、无法被攻破、100% 法律正确。

可信度来自公开威胁模型、代码、测试和审计，而不是形容词。

### 7.2 数据分级

#### Red —— 永不离开用户控制设备

主密钥、身份证件、客户完整名单、银行凭证、私人合同原件、API Secret、未发布商业机密。

#### Amber —— 经过本地脱敏后才可调用云模型

删除个人身份信息后的邮件、摘要化合同条款、匿名化业务数据、最小必要字段。

#### Green —— 可以发送给经过批准的模型

公开市场资料、通用文案、已公开产品信息、不含身份的数据。

每次模型调用必须生成 Data Disclosure Record：

```text
发送给了哪个供应商
使用了哪个模型
发送了哪些字段
数据属于什么等级
为什么需要发送
保留政策是什么
由哪项策略批准
```

### 7.3 密钥体系

```text
User Root Key
 └── Company Key
      └── Project Key
           ├── Database Key
           ├── File Key
           ├── Backup Key
           └── Audit Signing Key
```

要求：每家公司相互隔离、每个项目相互隔离、密钥支持轮换、设备丢失后可撤销、云端不持有完整根密钥、恢复机制不能依赖项目官方服务器、使用成熟密码库不自行设计算法。

### 7.4 零信任与最小权限

安全边界不应依据"它是官方 Agent""它运行在本地"或"用户安装过插件"而默认信任。

每次访问都应重新检查：主体是谁、操作是什么、资源是什么、数据是什么等级、当前设备是否可信、Token 是否过期、风险等级、是否需要用户批准。

## 八、Agent 和插件安全

### 8.1 插件默认不可信

第三方插件不得默认：

- 与核心进程共享内存
- 读取完整 Home 目录
- 访问所有环境变量
- 获取永久 API Key
- 自由连接任意域名
- 修改系统提示词
- 修改自己的权限
- 静默安装依赖
- 调用 Docker Socket
- 读取其他 Agent 的数据

### 8.2 插件清单（签名 Manifest）

```yaml
name: customer-email-drafter
version: 1.0.0
publisher: ...
permissions:
  data:
    - customer.display_name
    - customer.last_interaction_summary
  tools:
    - email.create_draft
  network:
    - none
  filesystem:
    - workspace:read
    - output:write
risk_level: low
signature: ...
```

安装界面向小白翻译为：

> 这个插件可以读取客户姓名和最近一次互动摘要，可以创建邮件草稿，但不能发送邮件、读取完整邮箱或访问互联网。

### 8.3 双重隔离

- **低风险插件**：运行在 WASM/WASI 沙箱中。
- **高风险工具**：运行在临时容器或微型虚拟机中（默认无网络、只挂载必要文件、临时凭证、CPU/内存/时间限制、任务完成后销毁、输出经过扫描）。

### 8.4 Capability Token

Executor 不获得永久权限，而只获得类似：

```text
允许：
在未来 5 分钟内
为客户 customer_123
创建一封邮件草稿
不得发送
不得读取其他客户
最多调用一次
```

Token 必须：短期有效、绑定具体资源、限定操作、可撤销、防重放、可审计、无法由 Agent 自行扩大。

## 九、自动网络安全防御

安全模块按照 NIST CSF 2.0 六项核心功能建立能力：Govern、Identify、Protect、Detect、Respond、Recover。

### 9.1 资产清单

系统维护：软件资产、云账户、设备、仓库、域名、API、数据库、OAuth 授权、Agent、MCP Server、插件、凭证、智能合约、外部依赖。

### 9.2 自动检测

检测范围：异常登录、API 调用激增、大量数据读取、Agent 越权、Prompt Injection、插件行为与 Manifest 不一致、凭证泄露、依赖漏洞、审计日志断链、备份长期未成功、节点状态分叉、模型输出敏感信息、异常资产转移请求。

### 9.3 自动响应权限

#### 可以自动执行

暂停可疑 Agent、撤销短期 Token、阻止数据外传、隔离可疑文件、将工具切换为只读、停止异常工作流、启动备用模型、切换恢复节点、保存事件证据。

#### 必须人工批准

删除业务数据、关闭生产业务、永久封禁客户、撤销根证书、对外披露事件、向监管机构报告、转移资金、销毁密钥。

**AI Security Analyst 只提出判断，Policy Engine 决定是否允许，Executor 使用受限 Token 执行。**

## 十、法务和税务功能

### 10.1 Jurisdiction Pack

每一个国家或地区由一个版本化司法辖区包支持：

```text
jurisdictions/
  singapore/
    entity-rules
    contract-rules
    privacy-rules
    tax-rules
    filing-calendar
    legal-sources
    test-cases
    escalation-policy
```

每条规则必须包含：司法辖区、来源、生效日期、最后验证日期、适用条件、所需事实、不确定性、是否需要专业人士确认。

### 10.2 法务输出结构

每一份法务分析都应包含：

1. 已知事实
2. 采用的假设
3. 可能适用的国家或地区
4. 风险点
5. 依据来源
6. 来源生效日期
7. 可执行建议
8. 需要律师确认的问题
9. 明确的不确定性

系统定位应是：**法律信息、文件草拟、合规管理与专业人员协作平台**。不能一开始宣称是"可以代替全球执业律师的 AI 律师"。

### 10.3 税务系统

税务计算应尽量由确定性规则完成：

```text
业务交易
→ 交易分类
→ 司法辖区判断
→ 税务规则匹配
→ 汇率和期间处理
→ 计算
→ 人工复核
→ 申报材料
```

LLM 用于：解释、信息抽取、异常识别、文件草拟。

LLM 不应独自决定：税率、申报义务、最终税额、不可逆税务选择。

首个版本只支持一个明确司法辖区。结合项目发起人的学习和生活背景，可以首先制作新加坡数字服务或咨询型一人企业 Pack，但正式标记为 "Verified" 之前必须有当地专业人士审阅。

## 十一、区块链和分布式资产

### 11.1 区块链不是核心数据库

用户个人信息、合同全文、客户资料和发票不得直接写入公共区块链。

合理结构：

```text
链下：加密业务数据
链上或透明日志：哈希、时间戳、签名、状态承诺、审计根
```

### 11.2 合理使用场景

合同版本存在性证明、审计日志根锚定、多方审批、数字资产对账、多签钱包、支出上限、智能合约托管、跨组织合规证明。

### 11.3 Agent 对资产的权限

Agent 可以：准备交易、模拟交易、解释风险、检查地址、计算资产影响、生成待签名请求。

Agent 不可以：持有完整私钥、绕过多签、修改白名单、独立转移资产、关闭审计记录。

数字资产能力应在核心系统稳定、经过安全审计和明确法律评估之后再加入。

## 十四、推荐技术栈

### 14.1 核心原则

- 信任边界内尽量减少语言数量
- 安全关键代码采用内存安全语言
- AI 与领域扩展可以使用 Python
- 所有组件通过稳定协议交互

### 14.2 推荐组合

#### Secure Core —— Rust

负责：密钥和加密、本地 Vault、策略执行、Capability Token、签名事件日志、插件沙箱、节点通信、故障恢复。

#### UI 与 SDK —— TypeScript + React + Tauri

负责：桌面端、Founder Command Center、插件 SDK、可视化审批、Sovereign Enterprise Graph UI。

#### Agent 与领域开发 —— Python（作为非信任边界内的隔离 Worker）

负责：Agent Workflow、RAG、法务和税务抽取、模型评测、数据分析、领域 Pack 工具。

**Python Worker 不应直接持有主密钥或永久权限。**

#### 协议

- JSON Schema：公共数据合同
- Protobuf/gRPC：内部高性能通信
- WIT/WASI：插件能力接口
- MCP：外部工具兼容
- A2A：未来跨 Agent 系统兼容

## 十七、安全测试与故障演练

### 17.1 必须通过的测试

#### 数据安全

- 模型只能看到明确批准的字段
- Red 数据不能进入云模型请求
- 被撤销设备无法解密新数据
- 恢复节点无法单独读取全部数据

#### 权限安全

- Agent 不能扩大自身 Capability
- Token 不能用于其他客户或项目
- Token 过期后立即失效
- 重放请求被拒绝

#### 插件安全

- 插件不能读取未声明文件
- 插件不能访问未批准域名
- 插件不能读取主进程环境变量
- 插件不能修改自身 Manifest
- 恶意依赖不能绕过安装策略

#### 分布式安全

- 主节点死亡后可以恢复
- 旧主节点恢复后不能产生双主写入
- 损坏副本能够被检测
- 丢失一个副本不造成数据丢失
- 恢复流程不依赖官方云服务

#### Agent 安全

- Prompt Injection 不能改变系统权限
- 第二模型不能绕过主策略
- 模型拒绝或宕机不影响数据访问
- 恶意文档只作为不可信数据处理

```text
sovereign chaos revoke-provider
sovereign chaos kill-primary-node
sovereign chaos corrupt-replica
sovereign chaos inject-malicious-skill
```

## 十九、治理与商业模式

### 19.1 开源原则

核心采用宽松且包含专利授权的开源许可证，例如 **Apache-2.0**。

原因：企业更容易采用、研究项目更容易引用、安全工具更容易成为基础设施、避免用户怀疑核心安全能力被人为限制。

通过商标政策保护官方名称和发行版，而不是通过关闭核心代码建立壁垒。

### 19.2 商业化内容

核心本地运行能力必须完整可用。可以收费的服务：

- Server-blind 加密同步
- 托管恢复节点
- Verified Jurisdiction Packs
- 专业律师和会计师协作网络
- 企业部署支持
- 安全审计
- 合规报告
- 高可用节点托管
- 经过验证的插件市场
- 机密计算推理

不能采用：出售用户数据、广告画像、强制上传业务数据、将安全更新放在付费墙后、让官方云成为唯一恢复方式。

## 二十、开发阶段与退出条件

### Stage 0：项目宪法

交付：Vision、Threat Model、Trust Boundaries、Non-goals、Architecture RFC、Security principles。

退出条件：任意贡献者能够解释系统保护什么、信任谁、不信任谁。

### Stage 1：安全内核

交付：Vault、Identity、Policy、Capability、Sandbox、Audit Ledger。

退出条件：恶意 Agent 在没有授权的情况下无法读取文件或执行外部动作。

### Stage 2：模型韧性

交付：统一模型接口、多供应商路由、本地模型、健康检测、自动降级。

退出条件：删除主模型配置后，核心工作流仍能继续。

### Stage 3：工作流恢复

交付：Checkpoint、Idempotency、Event Replay、Backup、Node Promotion。

退出条件：杀死主进程后，另一个节点能够从最后有效事件恢复。

### Stage 4：Founder OS Demo

交付：Sovereign Enterprise Graph、客户接入、Offer、合同草稿、发票草稿、Founder Command Center。

退出条件：陌生用户不阅读源码也能完成一个真实业务流程。

### Stage 5：Security Gauntlet

交付：攻击测试集、Chaos CLI、框架对比、公开报告。

退出条件：项目能够用可重复实验而不是宣传语证明安全差异。

### Stage 6：司法辖区包

交付：Jurisdiction schema、来源和日期追踪、确定性规则、专业审核流程、首个国家或地区 Pack。

退出条件：每一项结论都能追溯到规则、来源和假设。

### Stage 7：生态系统

交付：Plugin SDK、签名插件、Pack Registry、OpenClaw/MCP compatibility、治理组织。

退出条件：第三方无需修改核心仓库即可发布安全扩展。

## 二十三、最终建议

立项对象是完整的 **Sovereign Founder OS**，但不能同时开工所有功能。它的产品本质是：

> **一套帮助任何人创建并经营一人公司的 AI 操作系统，同时让数据、决策权和企业持续运营能力始终掌握在创业者自己手中。**

真正可能创造新类别的组合是：

1. 把创业方向、产品、获客、交付、财务、法务和日常经营连成一个闭环。
2. 用 Sovereign Enterprise Graph 保存真实企业状态，而不是把公司留在聊天记录里。
3. Crew Orchestrator 根据业务目标组建 AI Crew，普通用户不需要配置 Agent。
4. Agent 之间相互制约，插件默认不可信。
5. 数据和密钥真正由用户掌控，模型、节点和云服务都可替换。
6. 业务状态可验证、可恢复，企业法务、税务和安全有明确责任边界。
